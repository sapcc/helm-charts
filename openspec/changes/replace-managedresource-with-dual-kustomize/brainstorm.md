# Brainstorm: replace-managedresource-with-dual-kustomize

## Design Summary

Restructure `metal-operator-remote` deployment from a Gardener `ManagedResource`-driven pipeline (with Makefile pre-rendering and Secret-wrapped payloads) to a **dual kustomize + sequential `kubectl apply`** pipeline that targets the workerless cluster directly for **all** state — CRDs, RBAC, the workerless namespace, and the `ValidatingWebhookConfiguration`. The webhook URL rewrite is eliminated entirely: instead, an `ExternalName` Service in the workerless cluster's `system` namespace (reusing upstream's namespace placeholder) bridges the workerless `clientConfig.service` reference to the host cluster's actual `metal-operator-webhook-service`. This lets `kustomize` consume upstream's `config/webhook/` natively via Git URL ref with a single `ExternalName` patch — no pre-rendered files, no Makefile, no `wrap-managedresources.sh`. Two coordinated kustomize roots in this repo (`host/`, `remote/`) are consumed by parallel kube-secrets overlays (`host/`, `remote/`) and applied in two `kubectl apply -k` steps (remote first, host second). The webhook-injector sidecar narrows to **TLS cert + caBundle rotation only** — the workerless `ValidatingWebhookConfiguration` is the source-of-truth on the workerless cluster, and the sidecar reads/patches only its `caBundle` field. The pipeline is operated short-term by Concourse and is structurally compatible with mid-term Flux migration via two `Kustomization` CRs.

**Tracking**: implements the parent epic [`cc/unified-kubernetes#831`](https://github.wdf.sap.corp/cc/unified-kubernetes/issues/831) ("Refactor deployment automation for split host/remote cluster setup"). Continues the work of the archived POC change `2026-05-13-poc-kustomize-metal-operator-remote` (tracked as `cc/unified-kubernetes#1169`). The webhook-injector sidecar role is per `cc/unified-kubernetes#914` (production rollout). Scope of this change is **metal-operator-remote only**; other operators in #831 (boot, ipam, argora, khalkeon) are follow-up changes that can adopt the same pattern.

## Alternatives Considered

### Option A: Dual kustomize, narrow sidecar to caBundle-rotation, ExternalName-on-workerless (CHOSEN)

- **Approach**: Two kustomize roots (`host/`, `remote/`). The workerless cluster receives upstream's webhook config natively via `resources: <Git URL ref>` plus an `ExternalName` Service patch that bridges to the host cluster's actual `metal-operator-webhook-service`. The webhook-injector sidecar is narrowed to managing TLS cert lifecycle on host + patching `caBundle` on the workerless `ValidatingWebhookConfiguration`. Per-cluster overlay in `cc/kube-secrets` parameterizes the `externalName` to point at the seed-cluster namespace where the controller pod actually runs.
- **Pros**: Pure native kustomize end-to-end — zero pre-rendered files; upstream version bumps are absorbed automatically via Git URL ref. All six stated motivations satisfied (no Makefile, no Secret bulk, no commit cycle, native pipeline, debuggable, portable). Source-of-truth coherence — `kubectl diff -k remote/` shows webhook drift directly. The remaining sidecar role is small and well-bounded.
- **Cons**: Requires the SAP-internal webhook-injector binary to expose a "caBundle-rotation-only" mode (`WEBHOOK_INJECTOR_MODE=ca-rotation` env var or equivalent flag). Loses the sidecar's continuous reconciliation of the webhook config content (only caBundle is reconciled now); manual edits to the workerless `ValidatingWebhookConfiguration` are repaired only on next `kubectl apply -k` rather than continuously. Cluster teardown of webhook configs requires `kubectl delete -k remote/` rather than relying on sidecar absence.
- **Why chosen**: The architectural-correctness story is materially better, AND the upstream-tracking story is materially simpler. Webhook-injector binary owners are accessible and the mode change is small. The continuous-reconciliation loss is acceptable because manual edits to a workerless `ValidatingWebhookConfiguration` are extraordinary events that should be visible via PR-based reconciliation.

### Option B: Keep ManagedResource, drop only the Makefile

- **Approach**: Kustomize tree generates `ManagedResource` + `Secret` pairs **live** at build time (no committed pre-renders). Single `kubectl apply` to seed; gardener-resource-manager (GRM) still pushes to remote.
- **Pros**: Minimal architectural change. GRM keeps providing reconciliation, drift detection, ordered cleanup.
- **Cons**: Secret-wrapping bulk persists. Pipeline remains Gardener-only. Two-step apply expectation isn't met. Only one of six motivations addressed.
- **Why not chosen**: Directly contradicts the user's stated motivations.

### Option C: Dual kustomize, retain full webhook-injector (push role + caBundle), commit upstream-verbatim webhook YAML on host

- **Approach**: Same dual-kustomize structure as Option A, but the sidecar keeps its current "read host ConfigMap, push to workerless" role. To eliminate the URL rewrite, commit upstream's `manifests.yaml` verbatim at `host/base/webhook/upstream-webhooks.yaml`; `configMapGenerator` wraps it into ConfigMap `webhook-config`; `ExternalName` Service still goes on workerless to bridge `webhook-service.system` → host's actual Service.
- **Pros**: No webhook-injector binary change. Continuous reconciliation of webhook content via sidecar push. Less spec churn on `kustomize-sidecar-injection`.
- **Cons**: One pre-rendered file (`upstream-webhooks.yaml`) refreshed via `curl` on each upstream version bump. `configMapGenerator` cannot consume kustomize-build output, so this file must exist as a static commit. Webhook content's source-of-truth is split between host (ConfigMap content) and workerless (live state).
- **Why not chosen** (after correcting the DNS-resolution model): once we accepted that `ExternalName` belongs on workerless, the case for direct-apply-to-workerless became compelling enough to justify the binary change. The pre-render commit cycle, while bounded, contradicts the "native pipeline" motivation more strongly than the binary change does.

## Agreed Approach

**Option A** above. Architectural and operational details are documented in `design.md`. Summary:

1. Two cluster-targeted kustomize roots in this repo: `host/` (controller pod, services, sidecar in caBundle-rotation mode, supporting RBAC) and `remote/` (CRDs, RBAC, custom RBAC, Namespace, **ValidatingWebhookConfiguration via Git URL ref**, **`system` Namespace + `webhook-service` ExternalName Service**).
2. Webhook delivery to workerless: a two-layer kustomize structure. Inner layer (`remote/upstream/webhooks/upstream-no-svc/kustomization.yaml`) references `https://github.com/ironcore-dev/metal-operator//config/webhook?ref=v0.4.0` and applies `$patch: delete` to remove upstream's regular ClusterIP Service. Outer layer (`remote/upstream/webhooks/kustomization.yaml`) references the inner layer + adds the `system` namespace + our own `webhook-service-stub.yaml` (ExternalName Service). Robust to upstream file renames/additions; no pre-rendered file.
3. Webhook URL resolution path: workerless API server reads `clientConfig.service: webhook-service/system` → looks up Service in workerless etcd → finds `ExternalName` Service → resolves the externalName via the API server pod's `/etc/resolv.conf` (host cluster's CoreDNS) → reaches the host cluster's actual `metal-operator-webhook-service`.
4. Webhook-injector sidecar role narrows to: (a) generate/rotate TLS serving cert for the host's `metal-operator-webhook-service`, (b) patch `caBundle` on the workerless `ValidatingWebhookConfiguration` via remote-kubeconfig. No webhook content push, no local `webhook-config` ConfigMap on host.
5. Per-cluster overlays in `cc/kube-secrets` get a parallel `host/` + `remote/` subpath layout under `values/kustomize/<clusterType>/<region>/<clusterName>/metal-operator-remote/`. The per-cluster value is the **`externalName` field of the workerless `webhook-service` ExternalName Service** (the seed-cluster namespace where the controller pod's actual Service lives).
6. Apply order: `kubectl apply -k …/remote/` (workerless kubeconfig) → `kubectl apply -k …/host/` (host kubeconfig). After step 1, webhook configs exist on workerless with empty caBundle; after step 2, sidecar starts and populates caBundle.
7. Removed from this repo: `remote/upstream/{crds-and-rbac,webhooks}/managedresources.yaml`, `remote/upstream/webhooks/manifests-url-based.yaml`, `host/base/webhook-config.yaml` (no longer needed), `scripts/wrap-managedresources.sh`, `system/Makefile` regen targets, the `Role → ClusterRole` conversion patch in `remote/upstream/crds-and-rbac/kustomization.yaml`.
8. Concourse pipeline definition + per-cluster overlays + the `cluster-overlay-layout` capability spec changes all live in the coordinated `cc/kube-secrets` change.

## Key Decisions

- **Drop `ManagedResource`**: Motivated by avoiding Makefile pre-rendering, eliminating Secret-wrapping bulk, removing the pre-render commit cycle, supporting portability beyond Gardener, improving `kubectl diff` debuggability, and accepting two-step apply as a feature rather than a problem to hide.
- **Concourse short-term, Flux mid-term**: The kustomize tree is structurally compatible with both. `kustomize build <leaf-overlay>` is the entire build step in both worlds — no Concourse-only conventions, no post-processing, no KRM function plugins.
- **Workerless API server resolves `clientConfig.service` via its OWN etcd, NOT host DNS directly**: Critical model correction made during brainstorming. The API server queries its own cluster's Service objects; for regular ClusterIP Services this fails on a workerless cluster (no kube-proxy → no routable ClusterIP). For `ExternalName` Services the API server reads the externalName field and resolves it via standard DNS using the API server pod's resolver — which IS the host cluster's CoreDNS, since the pod runs in the host cluster.
- **ExternalName Service goes on the WORKERLESS cluster**: Following directly from the corrected resolution model. A Namespace `system` + `webhook-service` ExternalName Service in the workerless cluster bridges upstream's `clientConfig.service: webhook-service/system` reference to the host cluster's actual `metal-operator-webhook-service`. The `system` namespace is workerless-only and conflicts with nothing (workerless clusters are otherwise empty by default).
- **No URL rewrite, no pre-rendered webhook YAML; pull upstream directory + remove upstream Service + add own ExternalName**: The workerless cluster receives upstream's `ValidatingWebhookConfiguration` content natively via `resources: https://github.com/ironcore-dev/metal-operator//config/webhook?ref=v0.4.0` (Git URL ref to upstream's directory, tracking upstream's own `kustomization.yaml`). Upstream's directory also contains a regular ClusterIP Service that is the wrong shape for cross-cluster delivery, so we remove it via a two-layer kustomize structure: an inner layer (`upstream-no-svc/`) pulls upstream and applies `$patch: delete` to remove the Service; the outer layer references the inner layer plus our own `webhook-service-stub.yaml` (ExternalName Service in `system` namespace). The two-layer structure is required because kustomize loads all `resources:` before applying patches — a single-layer naive `pull-and-replace` would error on the duplicate Service identifier. This approach is **robust to upstream file renames and additions** (any new file upstream adds is automatically pulled in via the directory ref). Upstream version bumps propagate by changing `?ref=v0.4.0` → `?ref=v0.5.0` — no `curl + commit` cycle.
- **Apply order: remote → host**: CRDs + RBAC + Namespace + custom RBAC + `ValidatingWebhookConfiguration` + `system` namespace + ExternalName Service all land on workerless first. Controller pod + Services + sidecar come up second on host. Sidecar then patches `caBundle` on the workerless WebhookConfiguration. This avoids both controller `CrashLoopBackOff` alerts (vs. host-first) and the worst case of "remote has CRDs but no webhook validation, accepting invalid CRs without admission".
- **Narrow webhook-injector sidecar to caBundle-rotation-only**: The sidecar's role is restricted to (a) generate/rotate the TLS serving cert for the host's `metal-operator-webhook-service`, (b) patch `caBundle` field of the named workerless `ValidatingWebhookConfiguration`. No webhook content read on host, no webhook content push to workerless. RBAC on remote scoped to `get` + `patch` on `webhooks[*].clientConfig.caBundle` only. Local `webhook-config` ConfigMap on host is deleted.
- **Webhook-injector binary needs a mode switch**: The existing binary's mandatory `--webhook-config-name` flag previously pointed at a local ConfigMap; in the new role it points at the workerless `ValidatingWebhookConfiguration` to patch caBundle into. Behavior selected by `WEBHOOK_INJECTOR_MODE=ca-rotation` env var (or `--mode=ca-rotation` flag) to preserve backward compatibility for any other consumers. Binary change is an out-of-repo coordinated dependency tracked in this change's Open Questions.
- **TLS cert source on host = self-signed by webhook-injector**: The webhook-injector self-signs both the CA and the leaf serving cert, manages rotation, writes the leaf cert to a shared `emptyDir` volume mounted by both itself and the controller-manager container (so the controller's webhook server can read it), and patches the CA's `caBundle` into the workerless `ValidatingWebhookConfiguration` via remote-kubeconfig. **No cert-manager dependency, no external PKI integration.** This is the existing webhook-injector behavior and continues unchanged in the narrowed (caBundle-rotation) mode. Per `cc/unified-kubernetes#831`'s "Solved" section, webhook-injector is the agreed mechanism for cross-cluster webhook certificate management.
- **Kustomize tree MUST NOT emit `caBundle` in any webhook entry**: Maintenance invariant. The webhook-injector sidecar patches `caBundle` at runtime; subsequent `kubectl apply -k remote/` (client-side three-way merge or server-side apply) preserves the sidecar's value ONLY if the applied manifest never includes `caBundle`. Upstream's `config/webhook/manifests.yaml` already omits `caBundle`, so the verbatim Git URL ref is safe. Any future patch or transformer added to `remote/upstream/webhooks/` MUST NOT introduce a `caBundle` field — even an empty string would clobber the sidecar's patch on every apply, creating a webhook-failure window. This invariant is testable in the verify phase: `kustomize build remote/upstream/webhooks/` output filtered through `yq '.webhooks[].clientConfig.caBundle'` must produce zero matches.
- **Directory naming preserved (`host/`, `remote/`)**: They're already cluster-role-neutral. Documented mapping: `host/` ≡ workload cluster (where the controller pod runs), `remote/` ≡ workerless cluster (API server only). Keeping the names avoids touching every kube-secrets overlay's Git URL ref.
- **Kube-secrets subpaths: `host/` + `remote/`**: One parent directory `metal-operator-remote/` per cluster, with two subpaths matching the helm-charts dir names. Each subpath is a self-contained kustomize root (one Flux `Kustomization` CR target).
- **Per-cluster value: NONE — `externalName` is identical across all clusters**: The workerless `webhook-service` ExternalName Service has `externalName: metal-operator-webhook-service` (the **short name**, no namespace suffix). This mirrors today's pre-rendered URL `https://metal-operator-webhook-service:443/...` which is also a short name. Both work because the workerless API server pod runs in the same Gardener-managed seed namespace as the metal-operator controller pod (e.g., `shoot--metal--<cluster>`), so the API server pod's `/etc/resolv.conf` search paths resolve the short name to the seed cluster's actual Service. Identical externalName value for every cluster — **zero per-cluster customization** in helm-charts or kube-secrets for webhook routing.
- **Namespace picture across the two clusters** (verified against the existing helm chart, not introduced by this change): The HOST cluster runs the controller pod in `metal-operator-system` namespace — set externally at `helm install --namespace metal-operator-system` time today, and equivalently set by the per-cluster overlay's `namespace:` field in the dual-kustomize world. This is what the controller's `--manager-namespace=metal-operator-system` arg refers to (kubebuilder convention: "namespace the manager is running in"). The WORKERLESS cluster has three namespaces: (a) `metal-servers` — user-facing landing namespace for **namespaced** metal-operator CRs (BMCSecret, BIOSSettings, ServerClaim, ServerMaintenance, etc.; Server CRs are cluster-scoped and don't need a namespace); kept as-is from today's design; (b) `system` — NEW for the `webhook-service` ExternalName Service (matches upstream's clientConfig.service.namespace placeholder); (c) `kube-system` — host of the `metal-token-rotate` ServiceAccount referenced by existing `rbac.yaml`, always exists. The host kustomize tree (`host/base/kustomization.yaml`) does NOT pin a namespace; the per-cluster overlay in kube-secrets is expected to set `namespace: metal-operator-system` to match the controller's arg expectation, mirroring today's `helm install --namespace=metal-operator-system` workflow.
- **Drop `Role → ClusterRole` conversion**: This was a `ManagedResource`-era workaround. The helm chart's bundled `managedresources/crds-and-rbac.yaml` proves that un-converted upstream Roles + RoleBindings work in production — the conversion was a defensive choice introduced during the kustomize POC, not a parity preservation. Removing it is a reversion to the proven helm-chart behavior.
- **Restore manager `args` missed by the kustomize POC, consolidate all manager patches**: The kustomize tree (introduced in the archived `poc-kustomize-metal-operator-remote` change) inherits the manager Deployment from upstream `config/manager`, which only sets `--leader-elect`. The 6 SAP-specific args from the helm chart's `values-overrides.yaml` (`--mac-prefixes-file`, `--probe-image`, `--probe-os-image`, `--insecure`, `--registry-url`, `--manager-namespace`) were never patched in. Production is still on helm so this hasn't surfaced yet. This change adds the args AND consolidates the existing `manager-remote-patch.yaml` + `manager-webhook-patch.yaml` into a single `host/base/manager-patch.yaml` so all controller-manager Deployment customizations live in one reviewable place.
- **Concourse pipeline scope = kube-secrets change**: This change defines the kustomize-side contract (two roots, sequential apply order, no MR, ExternalName-driven webhook routing). The pipeline definition + per-cluster overlays + value sources are part of the coordinated `cc/kube-secrets` change, like the previous `move-cluster-overlays-to-kube-secrets` was.
- **Remove all pre-render machinery aggressively**: `managedresources.yaml` files (in both this repo and the helm chart's bundled `managedresources/`), `manifests-url-based.yaml`, `wrap-managedresources.sh`, all `regen-metal-operator-remote*` Makefile targets. With Option A there is also NO replacement pre-rendered file (unlike Option C which would have committed `upstream-webhooks.yaml`).

## Open Questions

- [ ] **OQ2 — webhook-injector binary mode switch**: The binary's current mandatory `--webhook-config-name` flag points at a local ConfigMap. The narrowed role needs it to refer to the workerless `ValidatingWebhookConfiguration` to patch. Implementation needs to add `WEBHOOK_INJECTOR_MODE=ca-rotation` env var (or `--mode=ca-rotation` flag) to the SAP-internal webhook-injector binary, with backward compatibility for any other consumers still using push mode. **Tracked at [SAP-cloud-infrastructure/webhook-injector#9](https://github.com/SAP-cloud-infrastructure/webhook-injector/issues/9)**. — owner: implementation phase, requires coordination with webhook-injector binary maintainers
- [ ] **Implementation-phase audit: `values-overrides.yaml` parity** — The helm chart has explicit overrides on `controllerManager.manager.{args, env, resources, volumes, securityContext}` and `controllerManager.{podSecurityContext, pod.labels, serviceAccountName, hostNetwork}`. Most are reflected in `host/base/manager-patch.yaml`, but pod-level Gardener network labels (`networking.gardener.cloud/to-dns: allowed` etc.) and `runAsUser: 65532` should be verified end-to-end. — owner: implementation phase
- [ ] **Implementation-phase audit: image tag, env, and other arg propagation** — With the manager `args:` gap addressed in scope (see Key Decisions), the remaining audit covers: per-cluster image tags in `cc/kube-secrets` overlays, env-var values like `KUBERNETES_SERVICE_HOST` (currently a placeholder `APISERVER_URL_PLACEHOLDER` in `manager-patch.yaml`), and any future per-cluster arg overrides (e.g., if `--registry-url` becomes per-cluster). Verify before cutover. — owner: implementation phase
- [ ] **Implementation-phase audit: any other GRM-era patches/transforms** — A thorough audit of the kustomize tree for any other ManagedResource-era workarounds beyond the `Role → ClusterRole` conversion. — owner: implementation phase
- [ ] **Behavior under network partitions** — If the sidecar can't reach the workerless API server (caBundle patch fails), what's the retry / alerting strategy? The existing sidecar reconciliation behavior may be sufficient; verify and capture in the spec or operational runbook. — owner: implementation / operational phase
- [ ] **Cluster teardown procedure** — When a cluster is decommissioned, `kubectl delete -k remote/` cleanly removes the workerless ValidatingWebhookConfiguration + Namespace + ExternalName Service. Verify this is the recommended teardown path; document it in the kube-secrets change. In the typical case the workerless cluster is destroyed wholesale so this is moot, but the explicit teardown command is a useful runbook addition. — owner: kube-secrets change / operational documentation
- [ ] **`webhook-url-rendering` capability becomes obsolete** — With the URL transform eliminated entirely (no rewrite, no pre-render), the existing `webhook-url-rendering` capability spec has nothing left to govern. Decision in the specs phase: remove the capability entirely, or rename it to something like `webhook-routing-via-externalname` to govern the new ExternalName-based bridging mechanism. — owner: specs phase (next OpenSpec artifact)
- [ ] **Gardener-managed token rotation persists** — The `serviceaccount.resources.gardener.cloud/name` annotations on `host/base/remote-serviceaccount-secret.yaml` and `host/base/rotate-kubeconfig-secret.yaml` keep this design coupled to GRM for **token rotation**, even though MR is removed for resource delivery. Per epic `cc/unified-kubernetes#831`'s "Solved" section, "Authentication is moving to structured auth (remote cluster trusts host cluster service account tokens), reducing dependency on GRM" — that migration is a separate workstream not blocked by this change. The dual-kustomize design is neutral: when structured auth lands, the two Secrets become plain Kubernetes Secrets (or are eliminated entirely) without requiring restructuring of the kustomize tree. — owner: separate change tracking the structured-auth migration
- [ ] **Multi-operator generalization deferred** — Epic `cc/unified-kubernetes#831` lists `metal, boot, ipam, argora, khalkeon` as operators sharing this pattern. This change is **metal-operator-remote only**. The dual-kustomize structure (`host/` + `remote/` roots, kustomize Git URL refs to upstream `config/manager` and `config/{crd,rbac,webhook}`, ExternalName-on-workerless for webhook routing, `configMapGenerator` for sidecar input where applicable) can be replicated for the other operators in follow-up changes. This design **does not extract reusable scaffolding** (e.g., a kustomize Component or a templated tree) — that's deferred to whichever change tackles operator #2 to avoid premature abstraction. — owner: future per-operator changes
- [ ] **webhook-injector production status verification for metal-operator-remote** — Issue `cc/unified-kubernetes#914` tracks production rollout of webhook-injector for `ipam-capi-remote, eu-de-3, eu-de-1, eu-de-2`. The current `components/webhook-injector/` directory and `_webhook-injector-sidecar.tpl` template indicate metal-operator-remote already uses the sidecar in some form, but: verify before merge that **all production metal-operator-remote deployments** (including `rt-eu-de-1`, `a-qa-de-200`) actually run the sidecar and that webhook configs on their workerless clusters today were created by the sidecar (not by some legacy path). If a legacy path exists, an additional migration step is needed before this change ships. — owner: implementation / operational verification
- [ ] **Deployment name mismatch (helm vs kustomize)** — The helm-rendered Deployment is named `metal-operator-controller-manager` (hardcoded in `templates/controller-manager.yaml` line 5); the kustomize-rendered Deployment is named `controller-manager` (inherited from upstream `config/manager` with no `namePrefix` in `host/base/kustomization.yaml`). Service selectors are label-based (`app.kubernetes.io/name: metal-operator`) so traffic isn't affected, but anything that references the Deployment **by name** — HPAs, PDBs, Prometheus rules / ServiceMonitors, runbooks, custom dashboards — will break at the helm→kustomize cutover. This is a pre-existing POC-stage decision (not introduced by this change) but the change should not ship before this is verified end-to-end. — owner: implementation / verify phase
